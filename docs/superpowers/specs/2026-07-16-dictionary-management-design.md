# 字典管理页面设计

## 背景与目标

在“三匠集团市场营销中台”的“系统管理”模块下新增“字典管理”页面。页面仅面向已经由后端分配该菜单权限的管理员，支持一级、二级字典的查询、新增、编辑和启停，不提供删除功能。

本次只修改 `vmp-front` 前端仓库，不修改或新增任何后端 Java 文件，也不修改数据库脚本。页面直接联调当前 Java 接口和 `vmp-api.postman_collection.json` 中定义的接口。

## 数据模型与接口约束

`sys_dict_data` 使用 `pid` 表达层级：

- `pid = VMP` 表示一级字典。
- 二级字典的 `pid` 等于所属一级字典的 `dictId`。
- `dictCode` 和 `dictName` 在数据库中均具有唯一索引。
- `status` 使用 `1` 表示启用，`0` 表示停用。

前端使用以下现有接口：

| 用途 | 方法与路径 | 参数/返回 |
| --- | --- | --- |
| 获取全部一级字典 | `GET /admin/loadAllFatherDictDataList` | 返回 `SysDictDataVO` 数组 |
| 获取二级字典 | `POST /admin/loadChildDictDataListByPid/{pid}` | 表单参数包含 `pageNo`、`pageSize`、可选的 `dictCodeFuzzy`、`dictNameFuzzy`、`status`；返回分页数据 |
| 新增一级字典 | `POST /admin/newDictData/VMP` | 表单参数包含 `dictCode`、`dictName`、可选的 `remark` |
| 新增二级字典 | `POST /admin/newDictData/{fatherDictId}` | 表单参数包含 `dictCode`、`dictName`、可选的 `remark` |
| 修改字典 | `POST /admin/updateDictData/{dictId}` | 一级提交 `dictName`、`remark`；二级额外提交 `status` |

修改接口会在后端忽略 `pid`、`dictCode` 和 `createTime`，因此修改弹窗中的 `dictCode` 只读展示，不发送该字段。

## 导航与路由

- 新增路由 `/system-management/dictionary-management`。
- 路由标题为“字典管理”，父级标题为“系统管理”。
- 后端数据库已经存在“字典管理”菜单记录，并已向管理员分配权限。前端继续使用现有动态菜单和路由权限校验，不注入虚拟菜单，不绕过权限判断。
- 在 `AppLayout.vue` 的图标映射中为该路径配置字典类图标。

## 页面布局

页面使用左右两栏布局：

- 左侧约占 70%，用于二级字典管理。
- 右侧约占 30%，用于一级字典管理。
- 小屏幕下两栏改为纵向排列，二级字典区域在上，一级字典区域在下。

### 左侧二级字典区域

顶部搜索栏必须使用 `el-row` 和 `el-col`，包含：

- 字典代码模糊查询输入框 `dictCodeFuzzy`。
- 字典名称模糊查询输入框 `dictNameFuzzy`。
- 状态下拉框 `status`，可选“启用”和“停用”。
- 查询、重置和新增二级字典按钮。

两个搜索输入框均支持按 Enter 触发查询。查询和重置会将页码恢复为第一页。

表格宽度为 100%，展示：

- 字典名称 `dictName`。
- 字典代码 `dictCode`。
- 状态 `status`，使用标签展示。
- 备注 `remark`。
- 操作列，提供“修改”按钮。

底部提供分页组件，默认每页 15 条，支持切换每页数量和页码。

未选中一级字典时，二级列表为空，“新增二级字典”按钮禁用，并展示引导用户选择一级字典的空状态。

### 右侧一级字典区域

顶部展示“新增一级字典”按钮，不提供条件查询。列表默认通过一级字典接口加载全部数据，每项只展示 `dictName`。

点击一级字典项后：

- 更新选中样式。
- 清空二级查询条件。
- 将二级列表页码恢复为第一页。
- 调用下级字典接口加载该一级字典的二级数据。

鼠标悬停一级字典项时，在该项末尾显示修改按钮。点击修改按钮不触发一级字典切换。

首次加载时默认选中第一项。一级字典新增或修改后重新加载一级列表，并优先保留操作前的选中项；若原选中项不存在，则选中第一项。一级字典为空时清空二级列表。

## 弹窗组件

新增独立组件 `src/components/dictionary-management/DictDataDialog.vue`，列表页不内嵌编辑弹窗。组件通过模式和目标字典支持四种场景：

- 新增一级字典。
- 新增二级字典。
- 修改一级字典。
- 修改二级字典。

### 新增模式

表单字段为：

- 字典代码 `dictCode`，必填。
- 字典名称 `dictName`，必填。
- 备注 `remark`，选填。

新增一级字典使用固定父级 `VMP`；新增二级字典使用当前选中一级字典的 `dictId`。后端统一将新增字典状态设为启用。

### 修改模式

- `dictCode` 使用禁用输入框展示原值，不允许修改。
- `dictName` 输入框保持空值，并使用原名称作为 placeholder。
- `remark` 输入框保持空值，并使用原备注作为 placeholder；原备注为空时展示通用选填提示。
- 修改二级字典时显示状态下拉框并展示当前状态；修改一级字典时不显示状态字段。

组件记录名称、备注是否被用户实际编辑。未编辑字段提交时使用原值；备注被编辑后清空则提交空字符串，从而支持清除备注。保存按钮仅在至少一个可编辑字段的最终值与原数据不同时可用。修改后的必填字段仍需通过校验。

所有弹窗在提交期间显示 loading 并阻止重复提交。请求失败时保留弹窗、输入内容和校验状态；成功后关闭弹窗并通知列表页刷新。

## 校验规则

在 `src/utils/Verify.js` 中增加公共校验器：

- `dictCode`：正则为 `^[A-Za-z0-9-]{1,20}$`，仅允许数字、英文字母和连字符 `-`，长度 1 至 20。
- `dictName`：正则为 `^[\u4e00-\u9fa5A-Za-z0-9-]{1,20}$`，仅允许中文、数字、英文字母和连字符 `-`，长度 1 至 20。

新增模式下两项均必填。修改模式下 `dictCode` 不参与提交；`dictName` 未输入时表示沿用原值，输入后必须通过公共校验器。`remark` 始终选填。

数据库唯一索引冲突等业务错误由后端返回，继续使用 `Request.js` 的统一错误提示。

## 状态与请求处理

- 一级列表、二级列表和弹窗提交分别维护独立 loading 状态。
- 二级列表请求只接受与当前选中一级字典 `dictId` 一致的响应，防止快速切换时旧响应覆盖新数据。
- 请求失败时不清空已有的成功数据；首次请求失败且没有缓存数据时展示空状态。
- 所有可选筛选参数仅在有值时发送，确保 `status = 0` 不会被误判为空。
- 接口返回值中的 `status`、分页数字统一转换为数字后再参与展示和计算。

## 验证策略

- 使用 Node 内置测试运行器为 `Verify.js` 的字典代码和字典名称校验补充单元测试，不引入新依赖。
- 执行前端测试，覆盖合法值、非法特殊字符、空值交由必填规则处理、20 位边界和超长值。
- 执行 `npm run build`，确认 Vue 模板、路由、SCSS 和导入均可构建。
- 页面联调时检查一级列表默认选择、父子切换、服务端分页、搜索 Enter、重置、新增两种层级、编辑 placeholder、无修改时禁用保存、备注清空、二级状态修改和失败保留表单。

## 完成标准

- 管理员可从“系统管理 / 字典管理”进入页面。
- 一级与二级字典严格按现有接口联动展示。
- 新增、修改、启停均可完成，且不出现删除入口。
- 页面与弹窗符合项目的 Vue、Element Plus、SCSS、命名和布局约束。
- 前端测试和生产构建通过，后端工作区不产生任何新增修改。
